# DGH 项目核心代码说明

## 1. 项目当前实际运行架构

当前项目实际运行的是一套：

- 后端：Spring Boot + MyBatis-Plus + MySQL + Redis
- 前端：Vue 3 + TypeScript + Vant + Vite

仓库中还保留了一套旧的 NestJS + SQLite 代码，路径主要在 `backend/src/*.ts`。这部分不是当前主系统运行链路，不应作为项目核心代码来理解。

当前主系统入口与配置：

- 后端启动入口：`backend/src/main/java/com/example/dgh/DghApplication.java`
- 后端配置：`backend/src/main/resources/application.yml`
- 数据库结构：`backend/src/main/resources/schema.sql`
- 前端入口：`frontend/src/main.ts`
- 前端全局壳层：`frontend/src/App.vue`

## 2. 项目核心业务目标

本项目是一个“设备报修/工单处理系统”，围绕三类角色展开：

- 普通用户：提交报修、查看自己的工单、给维修结果评价
- 维修人员：查看工单、签到、更新工单状态
- 管理员：分派工单、管理人员、查看统计和运营看板

系统围绕四类核心数据工作：

- 用户 `users`
- 地点 `locations`
- 工单 `tickets`
- 签到记录 `check_ins`

## 3. 后端核心代码

### 3.1 启动与基础配置

#### `backend/src/main/java/com/example/dgh/DghApplication.java`

功能：

- 启动 Spring Boot 应用
- 开启定时任务 `@EnableScheduling`
- 扫描 MyBatis Mapper `@MapperScan("com.example.dgh.mapper")`

作用：

- 它是整个后端服务的启动入口，决定了控制器、服务、Mapper、定时任务都能被 Spring 管理。

#### `backend/src/main/resources/application.yml`

功能：

- 配置服务端口 `3000`
- 配置 MySQL 数据源
- 配置 Redis
- 配置 MyBatis-Plus

作用：

- 决定后端连哪个数据库、监听哪个端口、是否启用 Redis 支撑验证码等功能。

#### `backend/src/main/resources/schema.sql`

功能：

- 定义核心表结构：
  - `users`
  - `locations`
  - `tickets`
  - `check_ins`

作用：

- 这是后端所有业务对象的数据基础。

### 3.4 工单主业务核心

#### `backend/src/main/java/com/example/dgh/service/TicketService.java`

这是整个项目最核心的业务服务类。

主要功能如下：

##### 1. 创建工单 `createTicket`

功能：

- 校验报修人是否存在且启用
- 校验地点是否有效
- 创建工单
- 根据环境变量决定是否自动派单

业务作用：

- 实现“用户提交报修”的核心流程。

##### 2. 查询工单列表 `listTickets`

功能：

- 查询所有工单或某个报修人的工单
- 按创建时间倒序
- 附加签到记录
- 补充报修人姓名、电话、地点名

业务作用：

- 支撑用户端、维修端、管理端的工单列表展示。

##### 3. 查询工单详情 `getTicket`

功能：

- 按工单 ID 查单条工单
- 附加签到和展示信息

业务作用：

- 支撑工单详情视图或后续操作前的装载。

##### 4. 更新工单状态 `updateStatus`

功能：

- 将工单更新为处理中或已完成

业务作用：

- 支撑维修员处理流程。

##### 5. 派单 `assignTicket`

功能：

- 给工单设置维修员姓名和联系方式
- 若工单原来是待处理，则转为处理中

业务作用：

- 支撑管理员派单流程。

##### 6. 签到 `checkIn`

功能：

- 为指定工单新增签到记录

业务作用：

- 支撑维修员到场签到。

##### 7. 提交评价 `submitFeedback`

功能：

- 保存工单评分和评价文字

业务作用：

- 支撑用户对维修服务的反馈。

##### 8. 基础统计 `stats`

功能：

- 统计总工单数
- 已完成工单数
- 平均评分
- 高频地点

业务作用：

- 给管理员首页统计卡片提供数据。

##### 9. 洞察摘要 `insightSummary`

功能：

- 计算高频区域
- 计算平均响应时长
- 统计夜间待处理/处理中工单

业务作用：

- 支撑首页运营摘要。

##### 10. 运营总览 `opsOverview`

功能：

- SLA 时限统计
- 超时与升级工单统计
- 区域工单分布
- 维修员绩效统计
- 高频问题统计
- 待评价工单统计

业务作用：

- 给管理员的运营看板提供核心分析数据。

##### 11. 超时工单查询 `findOverdueTickets`

功能：

- 查找超过指定小时数未完成的工单

业务作用：

- 用于定时提醒和运营分析。

### 3.5 认证与登录核心

#### `backend/src/main/java/com/example/dgh/service/AuthService.java`

功能：

- 用户注册
- 账号密码登录
- 手机验证码发送
- 手机验证码登录
- Redis 不可用时使用内存兜底保存验证码
- 生成登录态 token

业务作用：

- 它是账号体系的核心服务。

关键点：

- 管理员账号不允许通过普通注册创建
- 验证码有过期时间和发送冷却时间
- Redis 故障时不会导致整条验证码登录链路不可用

#### `backend/src/main/java/com/example/dgh/controller/AuthController.java`

功能：

- 对外暴露认证接口：
  - `POST /api/auth/register`
  - `POST /api/auth/login/password`
  - `POST /api/auth/login/code/send`
  - `POST /api/auth/login/code`

作用：

- 支撑前端登录页和注册页。

#### `backend/src/main/java/com/example/dgh/service/AuthTokenStore.java`

功能：

- 在内存中保存 token 与 userId 的对应关系

业务作用：

- 给登录后的前端返回一个可保存的登录凭证。

说明：

- 这是一个轻量实现，更适合演示系统，不适合高并发或分布式正式系统。

## 5. 核心业务流程

### 5.1 用户报修流程

流程：

1. 用户在 `LoginView.vue` 或 `RegisterView.vue` 登录/注册
2. 进入 `ReportView.vue`
3. 前端调用 `POST /api/tickets`
4. `TicketController` 收到请求
5. `TicketService.createTicket` 校验用户和地点并创建工单
6. 数据写入 `tickets` 表

### 5.2 用户查看与评价流程

流程：

1. 用户进入 `TicketsView.vue`
2. 前端调用 `GET /api/tickets?reporterId=...`
3. `TicketService.listTickets` 返回工单并附带签到、报修人、地点信息
4. 如果工单已完成，用户可调用 `POST /api/tickets/{id}/feedback`
5. `TicketService.submitFeedback` 保存评分和评价

### 5.3 维修员处理流程

流程：

1. 维修员登录进入 `TechnicianView.vue`
2. 前端获取工单列表
3. 维修员可调用：
   - `PATCH /api/tickets/{id}/status`
   - `POST /api/tickets/{id}/checkins`
4. `TicketService.updateStatus` 和 `TicketService.checkIn` 完成数据更新

### 5.4 管理员派单与运营流程

流程：

1. 管理员进入 `AdminView.vue`
2. 页面同时读取：
   - 工单列表
   - 基础统计
   - 运营总览
3. 管理员调用 `PATCH /api/tickets/{id}/assign`
4. `TicketService.assignTicket` 完成派单
5. 页面继续展示 SLA、超时、区域负责人、维修员绩效等分析数据

### 5.5 管理员人员管理流程

流程：

1. 管理员进入 `AdminUsersView.vue`
2. 调用 `GET /api/admin/users`
3. 新增人员时调用 `POST /api/admin/users`
4. 编辑人员时调用 `PATCH /api/admin/users/{id}`

## 7. 哪些代码不是当前主系统核心

以下内容不建议作为“当前项目核心代码”理解：

- `backend/src/*.ts`
- `backend/src/tickets/*.ts`
- `backend/src/entities/*.ts`

原因：

- 它们属于旧的 NestJS + SQLite 版本
- 当前项目实际启动和部署链路走的是 `backend/src/main/java/...`

## 8. 总结

这个项目的核心可以归纳为一句话：

- 后端以 `TicketService` 为中心，完成工单创建、流转、签到、评价、统计
- 前端以角色路由和四个主页面为中心，分别支撑用户报修、用户查单、维修处理、管理员派单和运营管理

若从答辩、汇报或写论文角度介绍项目，最应该重点讲的代码是：

- `TicketService.java`
- `AuthService.java`
- `AdminUserController.java`
- `ReportView.vue`
- `TicketsView.vue`
- `TechnicianView.vue`
- `AdminView.vue`
